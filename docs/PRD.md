# PRD: ГИС-САПР «Иннополис» — Единая геоинформационная платформа ОЭЗ

> **Версия**: 1.0  
> **Дата**: 22.03.2026  
> **Статус**: Draft  
> **Контекст**: хакатон, кейс ОЭЗ «Иннополис»

---

## 1. Резюме продукта

### 1.1 Проблема

ОЭЗ «Иннополис» охватывает три административных района Татарстана. Строительство продолжается, территория обслуживается несколькими отделами, данные разрозненны: кадастровые сведения — в Росреестре, проектная документация — у застройщиков, охранные зоны — в реестре ЗОУИТ, красные линии — в ГИСОГД. Ни один отдел не видит полной картины территории, а проектирование новых объектов требует ручной сверки множества ограничений из разных источников.

### 1.2 Источники проблемы

| Фактор | Описание |
|---|---|
| Устоявшаяся инфраструктура | Существующие инженерные сети, дороги и здания создают физические и логистические ограничения для нового строительства |
| Административная специфика | Три района — три набора документов, координация между отделами |
| Законодательные ограничения | Охранные зоны (ЗОУИТ), красные линии, ПЗЗ, нормативные отступы — десятки НПА федерального и регионального уровня |
| Инженерные ограничения | Пропускная способность сетей, сервитуты, требования к этажности и плотности застройки |

### 1.3 Решение

Геоинформационная САПР-платформа — единая система с картографической основой, которая:

- **Агрегирует** данные из Росреестра (НСПД), проектной документации, техпаспортов, ППТ и открытых источников в единую базу с семантической связью между графикой и атрибутами
- **Визуализирует** актуальное состояние территории ОЭЗ как многослойную карту
- **Валидирует** размещение новых объектов капитального строительства относительно нормативных, кадастровых и инженерных ограничений
- **Управляет** версиями проектных решений через модель сценариев

### 1.4 Ключевое отличие: бэкенд-first архитектура

Платформа спроектирована как GIS-движок/API, который прозрачно интегрируется с любым фронтендом — собственным веб-интерфейсом, существующим цифровым двойником Иннополиса или внешней ГИС (QGIS, ArcGIS, Cesium). На питче демонстрируется веб-прототип; для продакшена API интегрируется в 3D-систему заказчика.

## 2. Целевые пользователи и сценарии

### 2.1 Проектировщик (primary persona)

**Контекст**: специалист проектной организации или отдела капитального строительства ОЭЗ. Работает с генпланами, ситуационными планами, размещает объекты на территории.

**Ключевые сценарии**:

- Открывает карту, видит актуальное состояние территории: кадастровые участки, существующие здания, охранные зоны, красные линии — как переключаемые слои
- Выбирает тип объекта из каталога (здание, ТП, парковка), размещает на карте кликом или вводом координат (x, y, z), получает мгновенную визуальную обратную связь по ограничениям
- Перемещает/поворачивает объект до устранения всех конфликтов
- Загружает DXF от подрядчика: drag-and-drop → превью → выбор слоя → маппинг полей → отображение на карте
- Сохраняет размещение в рамках именованного сценария, сравнивает альтернативные варианты

**Принцип**: проектировщик не думает о координатных системах, не видит JSON, не знает про PostGIS. Интерфейс — как Cities: Skylines, но с реальными нормативами.

### 2.2 Сотрудник администрации ОЭЗ

**Контекст**: управляет территорией, контролирует процессы, отчитывается перед руководством.

**Ключевые сценарии**:

- Дашборд: количество участков (всего / свободных / строящихся), площади, статусы
- Фильтрация и выгрузка: «все объекты Лаишевского района со статусом строительство» → Excel/PDF
- Обогащение данных: добавление семантики к кадастровым объектам (арендатор, стадия стройки, фото)
- Отслеживание изменений из Росреестра: дифф — «три участка изменили границы с последней синхронизации»

**Права**: видит все слои и сценарии, редактирует семантику (статусы, комментарии, документы), не изменяет геометрию.

### 2.3 Системный администратор

**Ключевые задачи**: развёртывание через Docker Compose, бэкапы (pg_dump), мониторинг (healthcheck-эндпоинты), управление доступом (RBAC).

### 2.4 Разработчик-интегратор

**Ключевые задачи**: подключение новых источников данных (написание адаптеров), добавление типов правил валидации (SQL-генераторы), интеграция API в внешние системы (цифровой двойник).

## 4. Архитектура и технический дизайн

### 4.1 Принципы

1. **Backend-as-a-service**: все данные отдаются по OGC-стандартам (WFS, WMTS, OGC API Features). Любой GIS-фронтенд подключается без доработок.
2. **Registry-driven design**: новый слой, тип объекта, правило валидации — это запись в базе, не коммит. Расширение без деплоя.
3. **Единый SRID**: все данные хранятся в МСК-16 (зона 1). Пересчёт при импорте, отдача по запросу в нужной проекции.

### 4.2 Стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Хранилище | **PostGIS** (PostgreSQL) | Единая база: геометрия + семантика + реестры; ST_Transform, ST_Intersects, ST_Distance — вся валидация на стороне БД |
| Тайловый сервер | **pg_tileserv** | Go-бинарник от CrunchyData; автоматически публикует любую таблицу PostGIS как WMTS без конфигурации |
| Feature-сервер | **pg_featureserv** | Автоматическая публикация как OGC API Features; JSON/GeoJSON |
| Бизнес-логика | **FastAPI** (Python) | Валидация, импорт, сценарии, карточки; OpenAPI/Swagger из коробки |
| Демо-фронтенд | **MapLibre GL JS** | Открытый GL-рендер карт; заменяемый на любой другой фронт |
| Оркестрация | **Docker Compose** | `docker-compose up` — всё поднялось |
| Импорт геоданных | **GDAL/OGR** (ogr2ogr) | Стандарт де-факто; DXF, SHP, GeoJSON, GML, KML |
| Пересчёт координат | **pyproj** (API), **ST_Transform** (БД) | МСК-16 ↔ WGS-84 ↔ EPSG:3857 |

### 4.3 Структура проекта

```
/api            — FastAPI: роутеры, сервисы, модели, SQL-генераторы правил
/migrations     — Alembic: начальная схема + реестры слоёв/типов/правил
/docker         — docker-compose.yml + Dockerfiles для каждого сервиса
/docs           — ADR (Architecture Decision Records), Swagger (автогенерация)
/frontend       — MapLibre-приложение (демо)
/mock-data      — тестовые DXF, XML техпланов, GeoJSON охранных зон
```

### 4.4 Ключевые модели данных

**Реестр слоёв** (`layer_registry`): slug, name, geometry_type, source_srid, style (JSON), validation_rules (JSON), import_mapping (JSON).

**Каталог типов объектов** (`object_type_registry`): slug, name, default_size, setbacks (JSON), icon.

**Объекты** (`features`): id, layer_slug, geometry (PostGIS), properties (JSONB — произвольные key-value без миграций).

**Сценарии** (`scenarios`): id, name, author, status (draft/review/approved/rejected), created_at, parent_scenario_id.

**Дельты сценариев** (`scenario_features`): id, scenario_id, layer_slug, geometry, properties, action (add/modify/delete), base_feature_id.

**Документы** (`feature_documents`): id, feature_id, type (tech_plan/egrn/project/photo), file_path, metadata (JSONB).

**Связи** (`feature_relations`): id, source_feature_id, target_feature_id, relation_type (generates_zone/stands_on/belongs_to/replaces).

### 4.5 API-контракт (ключевые эндпоинты)

**Валидация размещения**

```
POST /api/v1/placement/validate
Request:  { geometry, object_type, params: { setback_m, fire_gap_m }, scenario_id? }
Response: { valid: bool, conflicts: [{ layer, feature_id, type, overlap_geometry, description }] }
```

**Карточка объекта**

```
GET /api/v1/features/{layer}/{id}/card
Response: { attributes, documents, relations, scenario_history }
```

**Импорт файлов**

```
POST /api/v1/import/upload
Request:  multipart (file + target_layer + source_srid?)
Response: { preview: { features_count, bbox, detected_fields }, import_id }

POST /api/v1/import/{import_id}/confirm
Request:  { field_mapping }
Response: { imported_count, warnings }
```

**Сценарии**

```
POST   /api/v1/scenarios                — создать
GET    /api/v1/scenarios/{id}           — получить с дельтой
PATCH  /api/v1/scenarios/{id}/status    — перевести по жизненному циклу
GET    /api/v1/scenarios/compare?a=X&b=Y — сравнение двух сценариев
```

### 4.6 Разделение ответственности: фронтенд / бэкенд

| Слой | Ответственность | Обоснование |
|---|---|---|
| Фронт (Turf.js, MapLibre Draw) | Перетаскивание, поворот, буферы отступов, визуальный фидбек | Мгновенный отклик, нельзя ждать 200 мс от API |
| Бэк (PostGIS + FastAPI) | Валидация пересечений, дистанций, вхождений; импорт; сценарии | Данные всех слоёв — только на сервере |

### 4.7 Типы правил валидации

| Тип | SQL-примитив | Пример |
|---|---|---|
| `no_intersect` | `ST_Intersects` | Нельзя строить в охранной зоне газопровода |
| `min_distance` | `ST_Distance` | 150 м до жилой застройки |
| `within_boundary` | `ST_Within` | Объект должен быть внутри границ участка |
| `max_coverage_ratio` | `ST_Area(intersection) / ST_Area(parcel)` | Процент застройки ≤ 40% |
| `along_red_lines` | `ST_Distance` до линий ППТ | Фасад по красной линии |

Реализация: конечный набор типов правил (SQL-генераторы, ~50 строк каждый); комбинация правил — декларативная конфигурация в реестре слоёв.

## 5. Источники данных

### 5.1 Подключаемые автоматически (на хакатоне)

| Источник | Данные | Доступ | Формат |
|---|---|---|---|
| **НСПД** (nspd.gov.ru) | Кадастровые участки, здания, ЗОУИТ — с геометрией | REST API + WMS, без ключей (публичная часть) | JSON, PNG-тайлы |
| **OpenStreetMap** | Дороги, здания, POI, landuse | Overpass API, без ключей | GeoJSON |

### 5.2 Мокируемые (демонстрация архитектуры интеграции)

| Источник | Данные | Реализация на хакатоне |
|---|---|---|
| **ЕГРН через СМЭВ** | Полные выписки, техпланы | XML-файл в mock-data → парсинг → база |
| **ГИСОГД Татарстана** | ППТ, красные линии, ПЗЗ | Тестовый слой в PostGIS |
| **Охранные зоны строящихся объектов** | Зоны, не внесённые в ЗОУИТ | Ручное создание, демонстрация валидации |
| **Проектная документация** | Генпланы, ситуационные планы | 1–2 тестовых DXF-файла |

### 5.3 Юридические основания

- НСПД: общедоступные сведения ЕГРН по 218-ФЗ, ст. 7, п. 5; справочный характер
- OSM: лицензия ODbL — свободное использование с указанием источника
- Открытые данные Татарстана (opendata.tatar.ru): свободное, бессрочное, безвозмездное использование
- Спутниковые снимки/подложка НСПД **не** являются общедоступными данными ЕГРН — используются только векторные данные реестра

## 6. Расширяемость: registry-driven design

Ключевой архитектурный принцип — расширение конфигурацией, а не кодом:

| Действие | Нужен код? | Что делать |
|---|---|---|
| Добавить слой | Нет | Запись в `layer_registry` → PostGIS-таблица создаётся автоматически |
| Добавить тип объекта | Нет | Запись в `object_type_registry` → фронт подхватывает динамически |
| Добавить правило из существующего типа | Нет | Конфигурация в `validation_rules` |
| Добавить новый *тип* правила | Да, минимально | SQL-генератор (~50 строк) + регистрация |
| Подключить новый источник данных | Да | Адаптер импорта → GeoJSON + метаданные → универсальный конвейер |

## 7. Нефункциональные требования

| Категория | Требование |
|---|---|
| **Развёртывание** | Одна команда `docker-compose up`; все сервисы контейнеризированы |
| **Документация API** | OpenAPI/Swagger автогенерация из FastAPI; является точкой интеграции с внешними системами |
| **Резервное копирование** | `pg_dump` по крону; единая база — единый бэкап |
| **Масштабирование** | Горизонтальное масштабирование FastAPI; PostGIS — вертикальное (реалистично для масштаба ОЭЗ) |
| **Безопасность** | RBAC через middleware/API-gateway; роли: проектировщик, администратор, сисадмин |
| **Производительность** | Валидация размещения < 500 мс; рендер карты — стандартный для WMTS |
| **Система координат** | МСК-16 зона 1 как внутренний SRID; поддержка WGS-84 и EPSG:3857 на вводе/выводе |

## 8. Метрики успеха

### 8.1 Для хакатона (демо)

1. Карта с реальными кадастровыми данными Иннополиса из НСПД + импортированный DXF-слой поверх
2. Интерактивное размещение объекта из каталога с валидацией: конфликт с охранной зоной → перемещение → «зелёный» статус
3. Карточка объекта: клик → атрибуты из разных источников в одном месте
4. «Как подключить к вашему двойнику?» — Swagger + OGC-эндпоинты; фронтенд заменяемый

### 8.2 Для продуктивного использования (горизонт)

- Время от получения проектной документации до отображения на карте: минуты вместо дней
- Количество источников данных, интегрированных в единую картину: ≥ 5
- Доля выявленных конфликтов размещения до начала стройки: → 100%
- Количество отделов ОЭЗ, работающих с единой базой: все

## 9. Что API отдаёт внешним потребителям

Платформа функционирует как GIS-движок, отдающий данные в стандартизированном виде:

- **Слои**: перечень и метаданные через OGC API Features
- **Тайлы**: растровые и векторные тайлы через WMTS (pg_tileserv)
- **Полигоны и объекты**: GeoJSON через OGC API Features (pg_featureserv) и REST API
- **Структурированные данные**: JSON-карточки объектов с атрибутами, документами, связями
- **Методы валидации**: REST-эндпоинт проверки размещения — геометрия на вход, список конфликтов на выход

Это позволяет на этапе продуктивной разработки интегрировать платформу в существующий цифровой двойник Иннополиса без замены фронтенда.

## 11. Ограничения и допущения

- **Цифровой двойник**: существует, но подключение к нему невозможно. Платформа проектируется для будущей интеграции через API; на хакатоне — собственный веб-интерфейс
- **СМЭВ / ГИСОГД**: на хакатоне недоступны — мокируются; архитектура адаптеров позволяет подключить при получении реального доступа
- **Мок-данные**: инженерные ограничения (пропускная способность сетей, сервитуты) генерируются как тестовые данные для демонстрации
- **Полнота ЗОУИТ**: реестр заполняется постепенно, не все зоны внесены — система поддерживает ручное добавление охранных зон
- **МСК-16**: формально параметры имеют гриф ДСП, фактически опубликованы; для ОЭЗ как государственной структуры использование МСК-16 — прямое требование кейса

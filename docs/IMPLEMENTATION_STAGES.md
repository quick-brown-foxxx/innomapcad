# Этапы реализации: ГИС-САПР Chrome Extension

> Статус: согласовано
> Дата: 2026-03-22
> Фокус: фундамент для автономной AI-разработки (тесты как контракт)

## Философия

Цель этапирования — не фичи, а **инфраструктура для AI-агентов**. Каждый этап заканчивается рабочими тестами, которые дают агенту обратную связь: сломал или нет.

- e2e тесты > unit тесты
- Реальный сайт (4dinno.ru) для e2e, чистая логика без браузера для unit
- Strict typing в обоих стеках (Python: basedpyright strict, TS: strict tsconfig)
- deck.gl v9 нативно типизирован — type narrowing через runtime guard на единственной `any`-точке (React fiber walk)

## Граф зависимостей

```
Этап 0 (скелет монорепы)
  ├── Этап 1 (backend)  ──┐
  └── Этап 2 (extension) ─┤  ← параллелятся
                           ├── Этап 3 (интеграция)
                           │     └── Этап 4 (размещение + валидация)
                           │           └── Этап 5 (полировка)
```

## Этап 0: Скелет монорепы + инфраструктура

**Зависимости:** нет, стартовая точка

**Что делаем:**
- Корневой `docker-compose.yml`, общие скрипты
- `backend/` — Python проект (uv, pyproject.toml, ruff, basedpyright strict, pytest)
- `extension/` — TypeScript проект (package.json, tsconfig strict, ESLint, Playwright)
- `tests/e2e/` — интеграционные e2e тесты (Playwright + backend)
- Pre-commit hooks, линтеры
- Единые скрипты для AI-агента: `test:all`, `lint:all`

**Результат:** AI-агент запускает `test` и `lint` в обоих проектах — зелёный ноль.

**Критерий готовности:**
- `cd backend && uv run pytest` — 0 тестов, 0 ошибок
- `cd extension && npm test` — 0 тестов, 0 ошибок
- `cd extension && npm run lint` — 0 ошибок
- `cd backend && uv run ruff check && uv run basedpyright` — 0 ошибок
- Pre-commit hooks проходят на пустом коммите

## Этап 1: Backend — данные + API + тесты

**Зависимости:** Этап 0
**Параллелится с:** Этап 2

**Что делаем:**
- FastAPI скелет: CORS, lifespan, `GET /health`
- GeoJSON данные: кадастровые участки + охранные зоны (реальные из НСПД или мок)
- `GET /api/v1/presets` — каталог 5 пресетов
- `GET /api/v1/layers/{slug}` — отдача GeoJSON (cadastral, protection_zones)
- `POST /api/v1/validate` — Shapely проверки: intersects, within, buffer+intersects (через pyproj UTM EPSG:32639)
- Dockerfile

**Тесты:**
- pytest + TestClient для каждого эндпоинта
- Реальный Shapely, реальные GeoJSON — никаких моков геометрии
- e2e: реальные HTTP-запросы к поднятому серверу

**Результат:** Все эндпоинты работают и покрыты тестами. AI-агент меняет код валидации — тесты показывают, сломал или нет.

**Критерий готовности:**
- `uv run pytest` — все тесты зелёные
- `curl localhost:8000/health` — `{"status": "ok"}`
- Каждый эндпоинт имеет минимум 2 теста (happy path + edge case)

## Этап 2: Extension — скелет + инжекция + тесты

**Зависимости:** Этап 0
**Параллелится с:** Этап 1

**Что делаем:**
- `manifest.json` (Manifest V3, content_scripts на `4dinno.ru/map/*`)
- `content.js` — ожидание `#deckgl-overlay`, инжекция page-context скриптов
- `injected.js` — получение Deck instance через React fiber + type guard
- `deck-bridge.ts` — патч `setProps`, управление кастомными слоями
- UI панель: базовый floating panel (content script context)

**Тесты:**
- **e2e (Playwright):** загрузка extension → навигация на реальный 4dinno.ru → проверка:
  - Panel видна в DOM
  - Deck instance найден (через `page.evaluate`)
  - `setProps` патч работает (слои добавляются)
- **Unit (без браузера):** чистая логика формирования слоёв, Turf.js валидация

**Playwright setup:**
- `launchPersistentContext` с `--load-extension`
- `channel: 'chromium'` для headless с extensions
- Тесты ходят на реальный 4dinno.ru (хакатон, 24 часа)

**Результат:** Extension загружается на реальном сайте, инжектит пустые слои, тесты зелёные.

**Критерий готовности:**
- `npm test` — unit тесты зелёные
- `npx playwright test` — e2e на реальном сайте проходят
- Extension видна в `chrome://extensions` (ручная проверка через Chrome DevTools MCP)

## Этап 3: Интеграция extension ↔ backend

**Зависимости:** Этапы 1 + 2

**Что делаем:**
- Extension загружает данные с бэкенда (`GET /api/v1/presets`, `GET /api/v1/layers/*`)
- Кадастровые участки рендерятся как GeoJsonLayer (синяя полупрозрачная заливка + обводка)
- Охранные зоны рендерятся как GeoJsonLayer (красная полупрозрачная заливка)
- UI панель: переключатели видимости слоёв, палитра пресетов

**Тесты:**
- e2e: `docker-compose up` (backend) + extension на реальном сайте →
  - Слои загружены (проверка через `page.evaluate` — layer IDs в deck instance)
  - Пресеты отображаются в панели
  - Переключение слоёв работает

**Результат:** Сквозной флоу данных: бэкенд → extension → deck.gl → карта.

**Критерий готовности:**
- e2e тест проходит с поднятым бэкендом
- На карте видны кадастровые участки и охранные зоны (визуальная проверка через Chrome DevTools MCP)

## Этап 4: Размещение объектов + валидация

**Зависимости:** Этап 3

**Что делаем:**
- Выбор пресета → клик по карте → прямоугольник по размерам пресета (SolidPolygonLayer extruded)
- Turf.js мгновенная валидация: пересечение с зонами, выход за границы участка
- Визуальный фидбек: зелёная обводка (ОК) / красная (конфликт) / оранжевый буфер отступа
- Кнопка «Проверить» → `POST /api/v1/validate` → серверный результат в панели
- Повторный клик — перемещение объекта

**Тесты:**
- **Unit:** Turf.js валидация — пересечение, непересечение, граничные случаи
- **e2e:** полный демо-сценарий из PRD:
  1. Открыть сайт → слои видны
  2. Выбрать пресет «Жилой дом»
  3. Кликнуть в охранную зону → красный статус
  4. Кликнуть на свободный участок → зелёный статус
  5. Нажать «Проверить» → серверный ответ отображается

**Результат:** Рабочее демо полного цикла.

**Критерий готовности:**
- Демо-сценарий из PRD воспроизводится автоматически через Playwright
- Unit-тесты Turf.js валидации зелёные
- Серверная валидация возвращает корректные конфликты

## Этап 5: Полировка + Chrome DevTools MCP

**Зависимости:** Этап 4

**Что делаем:**
- Обработка ошибок: бэкенд недоступен → degraded mode (только Turf.js)
- Обработка ошибок: deck.gl не найден → сообщение пользователю
- MutationObserver для устойчивости к React re-render
- Настройка Chrome DevTools MCP для ручного тестирования AI-агентом:
  - Загрузка extension в управляемом браузере
  - Навигация, клики, скриншоты через MCP tools
  - Проверка console.log на ошибки

**Результат:** Стабильное демо, устойчивое к ошибкам сети и ререндерам.

## Тестовая стратегия (сводка)

| Уровень | Инструмент | Что тестируем | Где |
|---------|-----------|---------------|-----|
| Unit (Python) | pytest + Shapely | Валидация геометрии, эндпоинты через TestClient | `backend/tests/` |
| Unit (TS) | Vitest | Turf.js валидация, формирование слоёв, утилиты | `extension/tests/unit/` |
| e2e (backend) | pytest + httpx | Реальные HTTP к поднятому серверу | `backend/tests/e2e/` |
| e2e (extension) | Playwright | Extension на реальном 4dinno.ru | `tests/e2e/` |
| e2e (интеграция) | Playwright + docker-compose | Backend + extension + реальный сайт | `tests/e2e/` |
| Ручное (AI) | Chrome DevTools MCP | Визуальная проверка, скриншоты, консоль | Этап 5 |

### Ключевые принципы тестирования
- **e2e на реальном сайте** — 4dinno.ru доступен, хакатон 24 часа, не мокаем
- **Unit без браузера** — чистая логика (Turf.js, Shapely) тестируется изолированно
- **Real > mocked** — реальный Shapely, реальные GeoJSON, реальный HTTP
- **Тесты = контракт для AI-агента** — если тесты зелёные, код работает

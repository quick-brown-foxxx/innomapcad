# ОЭЗ «Иннополис» — Единая геоинформационная платформа

## Суть проекта

Единая база данных об объектах ОЭЗ «Иннополис» с картографической основой, семантическими данными и инструментами проектирования. Территория ОЭЗ охватывает три административных района Татарстана; строительство продолжается.

Платформа решает три задачи: **видеть** актуальное состояние территории из разных источников на одной карте, **проверять** допустимость размещения новых объектов с учётом ограничений, **управлять** сценариями проектирования и их жизненным циклом.

## 1. Архитектура: ключевые решения

### 1.1 Бэкенд как сервис для любого фронтенда

> **Контекст**: у Иннополиса есть собственный фронтенд цифрового двойника неизвестного стека. Наш бэк должен работать с любым фронтом.

**Решение**: все данные отдаются по OGC-стандартам (WFS, WMTS, OGC API Features). Любой GIS-фронт — Cesium, ArcGIS, QGIS, кастомный — подключается из коробки.

**Стек**:

| Компонент | Технология | Роль |
|---|---|---|
| Хранилище | **PostGIS** | Единая база: геометрия + семантика + реестры |
| Тайлы | **pg_tileserv** | Автоматическая публикация слоёв как WMTS |
| Фичи | **pg_featureserv** | Автоматическая публикация как OGC API Features |
| Бизнес-логика | **FastAPI** | Валидация, импорт, сценарии, карточки объектов |
| Мок-фронт | **MapLibre GL JS** | Демо-интерфейс, заменяемый на любой другой |
| Оркестрация | **Docker Compose** | Один `docker-compose up` — всё поднялось |

pg_tileserv и pg_featureserv — легковесные Go-бинарники от CrunchyData, которые автоматически подхватывают любую новую таблицу в PostGIS без конфигурации. Это критично для расширяемости: добавил слой — он сразу доступен.

### 1.2 Registry-driven design: расширяемость без кода

> **Принцип**: всё описывается данными (реестрами), а не кодом. Новый слой, тип объекта, правило — это запись в базе, не коммит и не деплой.

**Реестр слоёв** (`layer_registry`) — описывает каждый слой: тип геометрии, стиль отображения, правила валидации, маппинг полей для импорта. Добавить слой = добавить запись; PostGIS-таблица создаётся автоматически.

**Каталог типов объектов** (`object_type_registry`) — размеры, класс, требуемые отступы, иконка. Фронт строит UI динамически из каталога.

**Правила валидации** — конечный набор *типов* правил, реализованных как SQL-генераторы; *комбинация* правил — конфиг. Как Terraform: провайдеры один раз, инфраструктура декларативно.

Пример записи реестра слоёв:

```yaml
- slug: "protection_zone_gas"
  name: "Охранные зоны газопроводов"
  geometry_type: polygon
  source_srid: 4326
  style: { fill: "#ff000033", stroke: "#ff0000" }
  validation_rules:
    - type: "no_intersect"
      target_layers: ["capital_buildings"]
      message: "Запрещено размещение ОКС в охранной зоне газопровода"
    - type: "min_distance"
      target_layers: ["residential"]
      distance_m: 150
  import_mapping:
    field_aliases:
      "НАИМЕНОВАНИЕ": "name"
      "КАД_НОМЕР": "cadastral_number"
```

Пример каталога типов объектов:

```yaml
- slug: "transformer_station"
  name: "Трансформаторная подстанция"
  default_size: [6, 4]
  setbacks:
    - to_layer: "residential"
      distance_m: 10
    - to_layer: "roads"
      distance_m: 5
  icon: "⚡"
```

**Уровни расширяемости**:

| Что добавляем | Нужен код? | Действие |
|---|---|---|
| Новый слой | Нет | Запись в `layer_registry` |
| Новый тип объекта | Нет | Запись в `object_type_registry` |
| Новое правило из существующего типа | Нет | Конфиг в `validation_rules` |
| Новый *тип* правила | Да, минимально | SQL-генератор (~50 строк) |
| Новый источник данных | Да | Адаптер импорта |

### 1.3 Система координат МСК-16

ОЭЗ работает в МСК-16 (местная система координат Татарстана). Параметры пересчёта из WGS-84/ГСК-2011 регистрируются как кастомный SRID в таблице `spatial_ref_sys` PostGIS. Пересчёт — через `ST_Transform` на стороне базы и `pyproj` на стороне API.

Делается один раз при развёртывании (SQL-миграция). Все слои хранятся в едином SRID, пересчёт происходит при импорте.

## 2. Источники данных и интеграции

### 2.1 Доступные программно (без авторизации)

| Источник | Что даёт | Как подключать |
|---|---|---|
| **ПКК Росреестра** | Кадастровые участки, здания, ЗОУИТ — с геометрией | REST API `pkk.rosreestr.ru/api/features/`, без ключей, по bbox или кад. номеру |
| **OpenStreetMap / Overpass** | Дороги, здания, инфраструктура | Overpass API, GeoJSON, без ключей |
| **Wikimapia** | «Народная карта» — описания объектов от пользователей | API `wikimapia.org/api`, возможный дополнительный слой |

**ПКК — главный автоматический источник.** Cron-job или ручная кнопка «обновить кадастр»: запрос по bbox территории ОЭЗ → сравнение с базой → дифф (изменения границ, ВРИ) → очередь на ревью.

### 2.2 Требуют доступа / мокируются

| Источник | Что даёт | Как подключать |
|---|---|---|
| **ЕГРН через СМЭВ** | Полные выписки, техпланы | Адаптер XML-схем (стандартизированы Росреестром). Для хакатона — мок: XML в папку → парсинг → база |
| **ИСОГД Татарстана** | ППТ, красные линии, ПЗЗ | Если есть WFS/WMS — внешний слой в реестре. Если нет — ручной импорт |
| **Проектная документация** | Генпланы, ситуационные планы | Импорт DXF/DWG/IFC через GDAL (`ogr2ogr`) |
| **Техпаспорта / техпланы** | Поэтажные планы, контуры | XML Росреестра или ручной импорт PDF (будущее — OCR) |
| **ППТ** | Красные линии, функциональные зоны | DXF — стандартный конвейер; PDF — на будущее |

### 2.3 Конвейер импорта (универсальный)

Три шага для любого файла:

1. **Формат → PostGIS**: `ogr2ogr` автоматически определяет формат (DXF, SHP, GeoJSON, GML, KML, MapInfo TAB) и загружает.
2. **Пересчёт координат**: SRID определяется из файла или указывается пользователем → `ST_Transform` к целевому SRID.
3. **Маппинг атрибутов**: `field_aliases` из реестра автоматически матчат поля. Незнакомые поля → UI «к чему привязать?».

Два пути ввода данных (оба проходят через одну валидацию и пишут в одни таблицы):

- **Импорт файла** — массовая операция, пакетная валидация, конфликты → отчёт.
- **Ручное добавление** — рисование на карте, форма атрибутов (генерируется из реестра), валидация в реальном времени.

## 3. Интерактивное размещение объектов

> **Метафора**: Cities: Skylines — выбрал объект из каталога, поставил на карту, увидел конфликты.

### 3.1 Разделение фронт / бэк

| Слой | Ответственность | Почему |
|---|---|---|
| **Фронт** (Turf.js) | Перетаскивание, поворот, буферы отступов, визуальный фидбек | Мгновенный отклик, нельзя ждать 200мс от API |
| **Бэк** (PostGIS) | Валидация пересечений, дистанций, вхождений | Данные всех слоёв — только на сервере |

### 3.2 Поток взаимодействия

1. Пользователь выбирает тип объекта из каталога (параметры: размеры, отступы).
2. Перетаскивает контур по карте — Turf.js мгновенно рисует буфер отступов.
3. «Бросает» объект — `POST /api/v1/placement/validate` с геометрией.
4. PostGIS за один запрос проверяет: `ST_Intersects` (охранные зоны), `ST_Within` (граница участка), `ST_Distance` (нормативные отступы), попадание в красные линии.
5. API возвращает список конфликтов с геометрией пересечений.
6. Фронт подсвечивает конфликтные зоны красным.

### 3.3 API-контракт

```json
// POST /api/v1/placement/validate
{
  "geometry": { "type": "Polygon", "coordinates": [...] },
  "object_type": "capital_building",
  "params": { "setback_m": 3, "fire_gap_m": 6 },
  "scenario_id": "uuid-optional"
}

// → 200 OK
{
  "valid": false,
  "conflicts": [
    {
      "layer": "protection_zones",
      "feature_id": "...",
      "type": "intersects",
      "overlap_geometry": { "type": "Polygon", "coordinates": [...] },
      "description": "Пересечение с охранной зоной газопровода"
    }
  ]
}
```

Тот же эндпоинт вызывается любым фронтом — нашим мок-интерфейсом или цифровым двойником Иннополиса.

### 3.4 Типы правил валидации (покрытие ~90% ограничений)

| Тип | SQL-примитив | Пример |
|---|---|---|
| `no_intersect` | `ST_Intersects` | Нельзя строить в охранной зоне |
| `min_distance` | `ST_Distance` | 150м до жилой застройки |
| `within_boundary` | `ST_Within` | Объект должен быть внутри участка |
| `max_coverage_ratio` | `ST_Area(intersection)/ST_Area(parcel)` | Процент застройки участка ≤ 40% |
| `along_red_lines` | `ST_Distance` до линий ППТ | Фасад по красной линии |

## 4. Версионирование: модель сценариев

> **Контекст**: проектирование идёт месяцы, варианты могут быть отменены, параллельные решения конфликтуют.

### 4.1 Почему не git-подобное ветвление

Мержить полигоны программно — нерешаемая задача в общем случае. Два проектировщика изменили границу одного участка — автоматический мерж невозможен. Лучше модель, близкая к реальному процессу проектирования.

### 4.2 Каноническое состояние + сценарии

**Каноническое состояние** — «как сейчас». Факт: кадастр, существующие здания, зафиксированные зоны. Обновляется только через импорт из авторитетных источников.

**Сценарий** — именованный набор изменений поверх канонического состояния. «Застройка квартала 3, вариант А», «Размещение ТП, вариант Б».

```
scenarios
  id, name, author, status (draft → review → approved | rejected),
  created_at, parent_scenario_id

scenario_features
  id, scenario_id, layer_slug, geometry, properties,
  action (add | modify | delete),
  base_feature_id  -- ссылка на оригинал если modify/delete
```

### 4.3 Что это даёт

- **Переключение**: карта + `?scenario_id=X` = каноническое состояние + дельта сценария.
- **Сравнение**: два сценария рядом — визуальный дифф.
- **Отмена**: сценарий → `rejected`, данные на месте, история сохранена.
- **Конфликты**: два сценария трогают один участок — автоматическая подсветка. Разрешение ручное (утвердить А, отклонить Б), не программный мерж.
- **Аудит**: каждое изменение — timestamp + автор.

## 5. Карточка объекта: семантика и связи

### 5.1 Модель данных

Каждый объект (feature) хранит:

- **Атрибуты** — JSONB `properties`: кадастровый номер, площадь, ВРИ, статус, произвольные key-value (как теги в OpenStreetMap — новый атрибут не требует миграции).
- **Документы** — `feature_documents`: техплан, выписка ЕГРН, проектная документация, фото. Тип + файл + метаданные.
- **Связи** — `feature_relations`: «этот газопровод → эта охранная зона», «здание → участок». Типы: `generates_zone`, `stands_on`, `belongs_to`, `replaces`.

### 5.2 Эндпоинт

`GET /api/v1/features/{layer}/{id}/card` — агрегирует всё: атрибуты, документы, связанные объекты, историю из сценариев. Структура карточки определяется реестром слоёв (какие поля, порядок, редактируемость).

### 5.3 Community-driven обогащение (референс Wikimapia)

Импортировали кадастр — там номер и площадь. Сотрудник администрации добавляет: «арендатор — ООО Рога», «стройка — фундамент готов», фото. Проектировщик прикрепляет ссылку на проект. Каждая роль обогащает данные по своей компетенции.

## 6. Пользовательские роли и сценарии

### 6.1 Проектировщик

**Основной пользователь.** Видит карту с актуальным состоянием территории (кадастр, здания, охранные зоны, красные линии — всё как слои с переключением). Выбирает объект из каталога, размещает на участке, система подсвечивает конфликты. Двигает/поворачивает до «зелёного» состояния. Сохраняет в рамках сценария.

Второй путь — загрузка DXF от подрядчика: drag-and-drop → превью → выбор слоя → маппинг полей → на карте.

**Не делает**: не думает о координатных системах, не видит JSON, не знает про PostGIS.

### 6.2 Сотрудник администрации ОЭЗ

**Контролирует и отчитывается.** Дашборд: сколько участков, свободных, строящихся. Карта — для визуализации, не редактирования.

Ключевые сценарии: фильтрация и выгрузка («все объекты Лаишевского района со статусом строительство» → Excel/PDF), отслеживание изменений из Росреестра (дифф: «три участка изменили границы»).

**Права**: видит всё, редактирует только семантику (статусы, комментарии, документы), не геометрию.

### 6.3 Сисадмин

**Docker Compose** (или Helm). Сервисы: PostGIS, pg_tileserv, pg_featureserv, FastAPI, фронт.

Задачи: бэкапы (`pg_dump` по крону, одна база), мониторинг (healthcheck-эндпоинты), управление доступом (RBAC через middleware или API-gateway), обновления (`docker-compose pull && up -d`).

Реестры в базе → обновление кода не затирает конфигурацию.

### 6.4 Разработчик (приёмщик проекта)

Открывает репозиторий:

```
/api          — FastAPI, стандартная структура
/migrations   — Alembic (начальная схема + реестры)
/docker       — compose + Dockerfiles
/docs         — Swagger (автогенерация) + ADR
/frontend     — MapLibre-приложение или заглушка
```

Добавить тип правила = один класс SQL-генератора + регистрация в маппинге. Подключить источник данных = адаптер → GeoJSON + метаданные → универсальный конвейер.

API документирован через OpenAPI/Swagger — точка интеграции с любым внешним фронтом или системой.

## 7. Оценка сложности и приоритеты

### Ядро (must have для хакатона)

| Компонент | Сложность | Комментарий |
|---|---|---|
| PostGIS + pg_tileserv + pg_featureserv | Низкая | Docker-образы, zero config |
| Реестр слоёв в базе | Низкая | Одна таблица + миграция |
| Импорт из ПКК Росреестра | Средняя | REST без авторизации, парсинг ответа, запись в PostGIS |
| Импорт файлов (DXF/GeoJSON) | Низкая | `ogr2ogr` — одна команда |
| Мок-фронт с картой | Средняя | MapLibre + подключение к pg_tileserv |
| Валидация размещения (API) | Средняя | SQL-генерация из конфига, 2–3 типа правил |

### Важно, но второй приоритет

| Компонент | Сложность | Комментарий |
|---|---|---|
| Интерактивное размещение (drag) | Средняя | Turf.js + MapLibre Draw, фронтовая работа |
| Каталог типов объектов | Низкая | Таблица + API для чтения |
| Карточка объекта | Низкая | Агрегирующий эндпоинт |
| МСК-16 | Низкая | Один SQL-insert в `spatial_ref_sys` |

### Продвинутое (wow-эффект / будущее)

| Компонент | Сложность | Комментарий |
|---|---|---|
| Сценарии проектирования | Средняя | Две таблицы + логика наложения дельты |
| Дифф обновлений из Росреестра | Высокая | Сравнение геометрий, UI конфликтов |
| Маппинг атрибутов через UI | Средняя | Фронт + бэк, UX-задача |
| СМЭВ / ИСОГД-интеграция | Высокая | Реальные доступы, которых нет на хакатоне |
| OCR / векторизация PDF ППТ | Очень высокая | R&D-задача |

## 8. Минимальный MVP для демо

Показать три вещи:

1. **Карта с реальными данными** — кадастровые участки Иннополиса из ПКК + слой из загруженного DXF поверх.
2. **Размещение объекта** — перетащил ТП из каталога, увидел конфликт с охранной зоной, подвинул.
3. **Карточка объекта** — клик → атрибуты из разных источников в одном месте.

Бонус для жюри: «Как это подключить к вашему двойнику? Вот Swagger, вот OGC-эндпоинты. Фронт заменяемый.»
